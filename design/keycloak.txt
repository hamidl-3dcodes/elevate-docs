Keycloak authentication 

Absolutely! Here’s a practical overview of how to authenticate and authorize users in a Paytech application using Keycloak, leveraging tokens (JWTs) for security.

⸻

🔐 High-Level Architecture Using Keycloak

Actors:
	•	Client Application (Mobile/Web App)
	•	Keycloak Server
	•	Backend Services (e.g., Payment Processor APIs)

⸻

✅ Key Features You’ll Use in Keycloak
	•	Realms: Separate security domains (e.g., one per environment or app)
	•	Clients: Represent your apps (e.g., mobile, backend)
	•	Users: Stored in Keycloak or external sources (LDAP, Identity Providers)
	•	Roles & Scopes: For authorization
	•	Token Types:
	•	Access Token (JWT): Used to access protected resources
	•	Refresh Token: Used to get a new access token
	•	ID Token: Contains user identity info

⸻

🛠️ Steps to Implement Authentication and Authorization

⸻

1. Set Up Keycloak
	•	Install Keycloak (or use a hosted solution)
	•	Create a Realm
	•	Create a Client (e.g., your mobile/web app)
	•	Set Access Type to confidential (for backend apps) or public (for frontend/mobile)
	•	Set Valid Redirect URIs and Web Origins
	•	Define User Roles (e.g., user, admin, merchant)
	•	Create Users (or allow registration)

⸻

2. Client-Side Authentication (Login Flow)

Use OAuth2 Authorization Code Flow with PKCE for mobile or SPA apps:
	1.	User clicks login → redirect to Keycloak login page
	2.	User authenticates → Keycloak redirects back with auth code
	3.	Client exchanges auth code for:
	•	Access Token (JWT)
	•	ID Token
	•	Refresh Token

You can use OIDC client libraries (e.g., oidc-client-js, AppAuth, etc.) to handle this securely.

⸻

3. Backend Token Verification

Your backend APIs should:
	•	Accept the Access Token in the Authorization: Bearer <token> header
	•	Validate the token:
	•	Signature (using Keycloak’s public key)
	•	Expiry
	•	Audience (aud)
	•	Scope/roles

You can use libraries like:
	•	Java: keycloak-spring-boot, spring-security-oauth2
	•	Node.js: keycloak-connect, express-jwt, jsonwebtoken
	•	Python: python-jose, FastAPI OAuth2, etc.

⸻

4. Authorization (RBAC)

Use Keycloak roles or scopes to enforce access:
	•	Define role mappings per client or realm
	•	Use role-based guards/middleware in your backend
	•	e.g., only merchant can initiate payments
	•	only admin can issue refunds

You can also use Keycloak’s Authorization Services for fine-grained access control (resource-based).

⸻

5. Token Refresh
	•	Access tokens are short-lived (e.g., 5–15 min)
	•	Use the refresh token to get new access tokens without re-authenticating

⸻

6. Logout and Revocation
	•	Implement single sign-out by redirecting to Keycloak’s logout endpoint
	•	Optionally invalidate refresh tokens

⸻

🧱 Optional Extras
	•	Enable MFA in Keycloak
	•	Set password policies, brute force protection
	•	Audit logs for logins
	•	Identity Brokering (e.g., login via Google, Apple)

⸻

Great! Here’s a backend validation example using Node.js + Express with JWT-based authentication and Keycloak.

We’ll validate:
	•	The token’s signature
	•	Its expiration
	•	Its audience (aud) and issuer (iss)
	•	Roles (authorization check)

⸻

🧰 Tech Stack
	•	Express.js
	•	jsonwebtoken (JWT validation)
	•	jwks-rsa (to fetch Keycloak public keys dynamically)

⸻

📦 1. Install Required Packages

⸻

⸻

🧩 3. middleware/auth.js — JWT Validation Middleware

const jwt = require('jsonwebtoken');
const jwksClient = require('jwks-rsa');
const dotenv = require('dotenv');

dotenv.config();

const client = jwksClient({
  jwksUri: `${process.env.KEYCLOAK_ISSUER}/protocol/openid-connect/certs`
});

function getKey(header, callback) {
  client.getSigningKey(header.kid, function (err, key) {
    const signingKey = key.getPublicKey();
    callback(null, signingKey);
  });
}

const validateToken = (requiredRoles = []) => {
  return (req, res, next) => {
    const authHeader = req.headers.authorization;

    if (!authHeader || !authHeader.startsWith('Bearer ')) {
      return res.status(401).json({ message: 'Missing or invalid Authorization header' });
    }

    const token = authHeader.split(' ')[1];

    jwt.verify(token, getKey, {
      audience: process.env.KEYCLOAK_AUDIENCE,
      issuer: process.env.KEYCLOAK_ISSUER,
      algorithms: ['RS256']
    }, (err, decoded) => {
      if (err) {
        return res.status(401).json({ message: 'Token verification failed', error: err.message });
      }

      // Optional: Role-based access check
      const roles = decoded.realm_access?.roles || [];

      const hasRequiredRole = requiredRoles.length === 0 ||
        requiredRoles.some(role => roles.includes(role));

      if (!hasRequiredRole) {
        return res.status(403).json({ message: 'Forbidden: insufficient role' });
      }

      req.user = decoded;
      next();
    });
  };
};

module.exports = validateToken;

🚀 4. server.js — Example API Routes

const express = require('express');
const validateToken = require('./middleware/auth');

const app = express();
const PORT = 3000;

app.get('/public', (req, res) => {
  res.send('This is a public endpoint');
});

app.get('/user', validateToken(['user']), (req, res) => {
  res.send(`Hello ${req.user.preferred_username}, you have USER access`);
});

app.get('/merchant', validateToken(['merchant']), (req, res) => {
  res.send(`Hello ${req.user.preferred_username}, you have MERCHANT access`);
});

app.listen(PORT, () => {
  console.log(`Server running on http://localhost:${PORT}`);
});

	•	Use HTTPS for all requests
	•	Enable CORS appropriately if frontend is on a different domain
